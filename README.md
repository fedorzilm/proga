# База Данных Интернет-Провайдера - Этап 4: Клиент-Серверная Архитектура

## 1. Введение

Данный документ описывает реализацию Этапа 4 проекта "База Данных Интернет-Провайдера".
Целью этого этапа являлось преобразование ранее созданного консольного приложения для работы с базой данных в полноценную клиент-серверную систему. Это включает разделение логики на серверную часть (обработка запросов, управление БД) и клиентскую часть (отправка запросов, получение результатов), а также организацию их сетевого взаимодействия по протоколу TCP/IP.

## 2. Реализованная функциональность на Этапе 4

### 2.1. Архитектура Клиент-Сервер

* **Сервер (`database_server`):**
    * Отвечает за хранение и управление базой данных абонентов (`Database`) и тарифным планом (`TariffPlan`).
    * Использует `QueryParser` для разбора текстовых запросов, полученных от клиентов.
    * Запускается как отдельное приложение, прослушивающее указанный TCP-порт для входящих клиентских соединений.
    * Обрабатывает каждого подключенного клиента в отдельном потоке (`std::thread`), обеспечивая базовую возможность одновременной работы нескольких клиентов.
    * Реализована синхронизация доступа к объекту `Database` с помощью `std::mutex` для предотвращения гонок данных при операциях записи (и для операций чтения для обеспечения строгой консистентности в текущей реализации).
    * Обрабатывает команды `ADD`, `SELECT`, `DELETE`, `EDIT`, `CALCULATE_CHARGES`, `PRINT_ALL`, `LOAD`, `SAVE`. Команды `LOAD` и `SAVE` выполняются на файловой системе сервера в специально отведенной и контролируемой директории (с проверкой безопасности путей для предотвращения выхода из "песочницы").
    * Корректно завершает работу при получении сигналов `SIGINT` / `SIGTERM` (для POSIX-систем) или консольных управляющих событий (для Windows), включая закрытие сокетов и попытку корректного завершения клиентских потоков.
    * Ведет логирование своей работы с использованием класса `Logger` с настраиваемым уровнем детализации.

* **Клиент (`database_client`):**
    * Запускается как отдельное консольное приложение.
    * Принимает адрес сервера, порт и другие опции через параметры командной строки.
    * Может работать в двух режимах:
        * **Интерактивный режим:** Пользователь вводит команды в консоль клиента, которые отправляются на сервер. Поддерживается локальная команда `HELP` и `QUIT_CLIENT`.
        * **Пакетный режим (`-f <файл_запросов>`):** Клиент считывает команды из указанного файла, отправляет их на сервер и сохраняет ответы сервера в выходной файл (имя формируется автоматически или указывается через опцию `-o`).
    * Отправляет команды на сервер и получает текстовые ответы, корректно обрабатывая их.
    * Реализована отправка специальной команды `EXIT_CLIENT_SESSION` серверу для корректного завершения сессии (например, при выходе из клиента или по команде `EXIT` из файла/интерактивного режима).
    * Ведет логирование своей работы с использованием класса `Logger`.

* **Сетевое взаимодействие:**
    * Осуществляется по протоколу TCP/IP.
    * Реализован протокол обмена сообщениями "длина (4 байта, network byte order) + данные (текстовая строка)". Это позволяет корректно обрабатывать сообщения переменной длины.
    * Класс `TCPSocket` инкапсулирует низкоуровневую работу с сокетами для Windows (Winsock) и POSIX. Включает:
        * Корректную инициализацию и очистку WSA для Windows (с подсчетом ссылок).
        * Создание, подключение (клиент), привязку и прослушивание (сервер), прием соединений.
        * Надежную отправку (`sendAllData`) и получение (`receiveAllData`) полных блоков данных, включая обработку частичных операций и `EINTR` для POSIX.
        * Реализацию методов `sendAllDataWithLengthPrefix` и `receiveAllDataWithLengthPrefix` с базовой поддержкой таймаута для операций получения (`SO_RCVTIMEO`).
        * Корректное закрытие сокетов с использованием `shutdown`.
        * Логирование всех значимых операций и ошибок.

### 2.2. Обработка команд на сервере

* Создан класс `ServerCommandHandler`, который инкапсулирует логику обработки всех типов запросов, отделяя ее от сетевого кода сервера.
* Логика обработки команд полностью адаптирована из класса `UserInterface` предыдущего этапа:
    * Все команды (`ADD`, `SELECT`, `DELETE`, `EDIT`, `CALCULATE_CHARGES`, `PRINT_ALL`, `LOAD`, `SAVE`, `HELP`, `EXIT`) имеют свои обработчики.
    * Операции `LOAD` и `SAVE` работают с файловой системой сервера. Реализована функция `getSafeServerFilePath_SCH` для формирования безопасных путей к файлам в пределах заданной "песочницы" (поддиректории `DEFAULT_SERVER_DATA_SUBDIR` относительно определенного корня данных сервера). Это предотвращает несанкционированный доступ к файловой системе. `Database::currentFilename_` теперь хранит полный канонический путь.
    * Результаты выполнения команд и сообщения об ошибках формируются в виде строки и отправляются клиенту.
    * Обеспечена подробная обработка исключений на уровне `ServerCommandHandler::processCommand`.

### 2.3. Логирование

* Реализован и интегрирован статический класс `Logger` (`logger.h`, `logger.cpp`) с поддержкой уровней логирования (DEBUG, INFO, WARN, ERROR, NONE) и выводом в консоль (и опционально в файл, хотя файловый вывод в `Logger::init` был закомментирован для простоты текущей итерации).
* Формат лога включает временную метку (с миллисекундами), уровень, ID потока, опциональное имя модуля и само сообщение.
* Вывод логов синхронизирован с помощью `std::mutex` для безопасного использования в многопоточной среде сервера.
* Логирование добавлено во все ключевые компоненты: `TCPSocket`, `Server`, `ServerCommandHandler`, `Client`, `Database`, `TariffPlan`, `QueryParser`, `FileUtils` и основные точки входа (`server_main.cpp`, `client_main.cpp`).

### 2.4. Обработка сигналов и Завершение работы

* **Сервер:** На POSIX-системах корректно обрабатывает сигналы `SIGINT` (Ctrl+C) и `SIGTERM` с использованием `sigaction` для штатного завершения. На Windows используется `SetConsoleCtrlHandler`.
* Глобальный флаг `g_server_should_stop` координирует процесс остановки.
* `Server::stop()` обеспечивает закрытие слушающего сокета, попытку корректного завершения (join) всех активных клиентских потоков и освобождение ресурсов.
* **Клиент:** Корректно завершает сессию с сервером при выходе или по команде `EXIT`.

### 2.5. Структура Проекта и Сборка

* Проект реорганизован с использованием поддиректорий в `src/` (`core`, `utils`, `net`, `server`, `client`) для лучшей модульности.
* `CMakeLists.txt` обновлен для сборки:
    * Статических библиотек `core_utils_lib` (основная логика, утилиты) и `net_lib` (сетевой код).
    * Исполняемых файлов `database_server` и `database_client`, линкующихся с этими библиотеками.
    * Утилиты `generate_data`.
    * Модульных тестов `run_unit_tests`, которые также линкуются с необходимыми библиотеками.
* Сохранены и обновлены цели для Doxygen, интеграционных тестов и анализа покрытия кода.

## 3. Что необходимо доделать и улучшить (для идеального соответствия Этапу 4 и дальнейшего развития)

### 3.1. Надежность и Производительность Сервера:

1.  **Пул потоков (Thread Pooling) для клиентов:** Вместо создания нового потока на каждое соединение, использовать пул потоков фиксированного или динамического размера. Это ограничит потребление ресурсов и накладные расходы при большом количестве одновременных клиентов.
2.  **Асинхронная сетевая модель (Advanced):** Для максимальной производительности и масштабируемости рассмотреть переход от модели "поток на клиента" к асинхронной модели на основе мультиплексирования ввода-вывода (например, `epoll` для Linux, `kqueue` для BSD/macOS, `IOCP` для Windows) или с использованием библиотек типа Boost.Asio или libevent. Это фундаментальное изменение архитектуры.
3.  **Более гранулярные блокировки БД (`std::shared_mutex`):** Заменить `std::mutex db_mutex_` на `std::shared_mutex` в классе `Server`. Операции чтения (SELECT, PRINT\_ALL, CALCULATE\_CHARGES) должны использовать `std::shared_lock`, а операции записи (`ADD`, `DELETE`, `EDIT`, `LOAD`, `SAVE`) — `std::unique_lock`. Это позволит нескольким клиентам одновременно выполнять операции чтения.
4.  **Таймауты на стороне сервера:** Реализовать таймауты не только на чтение от клиента (`SO_RCVTIMEO`), но и общие таймауты на длительность сессии неактивного клиента.
5.  **Ограничение ресурсов на клиента/запрос:** Предотвращение злоупотреблений (например, слишком большие запросы/ответы, слишком частые запросы от одного клиента).

### 3.2. Протокол и Данные:

1.  **Версионирование протокола:** Если протокол обмена будет усложняться, предусмотреть механизм версионирования.
2.  **Структурированный формат данных:** Вместо передачи всего как простого текста, рассмотреть JSON или бинарные форматы (Protocol Buffers, MessagePack) для сложных ответов или если планируется взаимодействие с клиентами на других языках. (Текущее ТЗ — текст).
3.  **Сжатие данных:** Для уменьшения трафика при передаче больших объемов данных (например, результатов `PRINT_ALL`).

### 3.3. Безопасность:

1.  **SSL/TLS шифрование:** Обязательно для защиты данных, передаваемых по сети, если приложение будет использоваться в недоверенной среде.
2.  **Аутентификация и Авторизация:** Если предполагается многопользовательский доступ с разными правами (выходит за рамки текущего ТЗ).
3.  **Усиление защиты "песочницы" для `LOAD`/`SAVE`:** Более строгие проверки путей, логирование всех файловых операций, возможно, белые списки разрешенных имен файлов или масок.

### 3.4. Клиентская часть:

1.  **Улучшенный интерактивный режим:** История команд, редактирование ввода (например, с использованием GNU Readline или аналогов).
2.  **Более детальная обратная связь клиенту:** Например, при ошибках сети или сервера.
3.  **Возможность отмены длительных операций** (сложно реализуемо с текущей синхронной моделью).

### 3.5. Тестирование:

1.  **Полное покрытие Unit-тестами:** Написать тесты для `TCPSocket` (включая специфичные для платформы ошибки и поведение), `ServerCommandHandler` (все команды с разными входными данными), `Logger`, `FileUtils`.
2.  **Комплексные Интеграционные тесты:**
    * Сценарии с большим количеством одновременных клиентов (нагрузочное тестирование).
    * Сценарии, проверяющие корректность работы `std::shared_mutex` (после его внедрения).
    * Сценарии с имитацией различных сетевых проблем (разрыв соединения, задержки).
    * Сценарии длительной работы сервера (soak tests) для выявления утечек ресурсов.
    * Автоматизированная проверка логов сервера и клиента в рамках тестов.
3.  **Тесты производительности (бенчмарки):** Для измерения пропускной способности сервера и времени отклика.

### 3.6. Качество кода и сборка:

1.  **Статический анализ кода:** Использование инструментов типа Clang-Tidy, Cppcheck для выявления потенциальных проблем.
2.  **Динамический анализ кода:** Использование Valgrind (на Linux) для обнаружения утечек памяти и ошибок работы с памятью. AddressSanitizer, ThreadSanitizer (ASan, TSan) при компиляции.
3.  **Конфигурационный файл для сервера:** Вынести все настраиваемые параметры (порт, путь к данным, путь к тарифу, уровень лога, параметры пула потоков и т.д.) в конфигурационный файл.
4.  **Кроссплатформенность:** Тщательное тестирование на всех целевых платформах (Windows, Linux, macOS). Убедиться, что платформозависимый код в `TCPSocket` и обработке сигналов работает корректно.
