# Проект "База Данных Интернет-Провайдера"

## Описание Проекта

Данный проект представляет собой систему управления базой данных для вымышленного интернет-провайдера. Система включает в себя:
* Модельную базу данных для хранения информации об абонентах, их IP-адресах, датах активности и почасовом трафике.
* Простейший язык запросов для взаимодействия с базой данных (выборка, добавление, удаление, редактирование записей, а также расчет счетов).
* Реализацию этой базы данных в виде сетевого приложения с клиент-серверной архитектурой, использующей собственный протокол обмена.
* Веб-интерфейс для работы с базой данных через стандартный веб-браузер по протоколу HTTP.
* Инструмент для генерации тестовых данных.
* Набор юнит-тестов и системных тестов для проверки корректности и надежности всех компонентов.

Работа с базой данных осуществляется в режиме "запрос-ответ". Система спроектирована с учетом возможности одновременного подключения нескольких независимых клиентов к серверу и включает механизмы для обеспечения корректности данных при конкурентном доступе. Особое внимание уделено модульности компонентов для их независимой модификации.

## Сборка Проекта

Проект использует CMake для сборки. Для компиляции требуется компилятор C++17 (например, GCC 8+ или Clang 7+) и Google Test (скачивается автоматически через `WorkspaceContent`).

1.  Создайте директорию для сборки (например, `build`) и перейдите в нее:
    ```bash
    mkdir build
    cd build
    ```
2.  Сконфигурируйте проект с помощью CMake:
    ```bash
    cmake .. 
    # или cmake -DCMAKE_BUILD_TYPE=Debug .. для отладочной сборки
    # или cmake -DCMAKE_BUILD_TYPE=Release .. для релизной сборки
    ```
3.  Скомпилируйте проект:
    ```bash
    cmake --build .
    # или make -jN (где N - количество ядер)
    # или ninja (если используется генератор Ninja)
    ```
Исполняемые файлы будут находиться в директории `build/bin/`, статические библиотеки (если создаются) в `build/lib/`.

## Компоненты и Исполняемые Файлы

### 1. `provider_app_standalone` (Монолитное приложение)
Консольное приложение для работы с базой данных в локальном режиме.

**Запуск:**
`./provider_app_standalone [<путь_к_файлу_БД> [<путь_к_файлу_тарифов>]]`
Или в пакетном режиме:
`./provider_app_standalone --input <файл_запросов> --output <файл_результатов> [--dbfile <путь_БД>] [--tariff <путь_тарифов>]`

* Если пути к файлам БД или тарифов не указаны, используются значения по умолчанию (`data/provider_data.txt`, `data/tariff.cfg`).
* **Интерактивный режим:** Предоставляет меню для выполнения операций: добавление, выборка, удаление, редактирование записей, расчет счета, загрузка тарифа, сохранение БД.
* **Пакетный режим:** Читает команды из `<файл_запросов>` и записывает результаты в `<файл_результатов>`.

### 2. `db_server_app` (Серверное приложение)
Сетевой сервер, который принимает подключения от клиентов и обрабатывает их запросы к базе данных. Поддерживает как кастомный протокол, так и HTTP.

**Запуск:**
`./db_server_app [--port <номер_порта>] [--dbfile <путь_к_файлу_БД>] [--tariff <путь_к_файлу_тарифов>]`

* `--port`: Номер порта для прослушивания (по умолчанию: 12345).
* `--dbfile`: Путь к файлу базы данных (по умолчанию: `data/provider_data.txt`).
* `--tariff`: Путь к файлу основного тарифного плана (по умолчанию: `data/tariff.cfg`).

Сервер обрабатывает конкурентные подключения от нескольких клиентов. При завершении работы (например, по сигналу SIGINT/SIGTERM) сервер сохраняет текущее состояние базы данных.

### 3. `db_client_app` (Клиентское приложение)
Консольный клиент для взаимодействия с `db_server_app` по кастомному сетевому протоколу.

**Запуск:**
`./db_client_app [--host <имя_хоста>] [--port <номер_порта>] [--id <id_клиента>]`
Или в пакетном режиме:
`./db_client_app --input <файл_сценария> --output <файл_результатов> [--host <хост>] [--port <порт>] [--id <id>]`

* `--host`: Имя хоста или IP-адрес сервера (по умолчанию: `127.0.0.1`).
* `--port`: Порт сервера (по умолчанию: 12345).
* `--id`: Идентификатор клиента (генерируется случайно, если не указан).
* **Интерактивный режим:** Позволяет вводить команды вручную.
    * Специальные команды:
        * `PING`: Проверка связи с сервером.
        * `ADD_INTERACTIVE`: Интерактивный ввод данных для новой записи.
        * `EDIT_INTERACTIVE`: Интерактивный ввод данных для редактирования записи.
        * `SHUTDOWN_SERVER`: Отправка команды на остановку сервера (требует соответствующих прав или конфигурации на сервере).
        * `EXIT`: Выход из клиента.
* **Пакетный режим:** Выполняет команды из `<файл_сценария>`.
    * Поддерживает команду `SLEEP <миллисекунды>` для вставки задержек.
    * Поддерживает команду `RAW_EDIT_COMMAND <payload_для_EDIT_RECORD_CMD>` для отправки команды редактирования с кастомным типом `EDIT_RECORD_CMD`.

### 4. `generate_data` (Генератор данных)
Утилита для создания файла с тестовыми данными абонентов.

**Запуск:**
`./generate_data <количество_записей> [<путь_к_выходному_файлу>]`

* Если путь к выходному файлу не указан, используется `data/provider_data.txt`.

## Язык Запросов (для Standalone и Кастомного Протокола)

### 1. `SELECT`
`SELECT <список_полей_или_*> [WHERE <условия>] [SORT BY <поле_сортировки> [ASC|DESC]]`
* `<список_полей_или_*>`: `*` для всех полей, или через запятую имена полей (например, `name, ip, date, total_traffic, total_incoming, total_outgoing, traffic_00, traffic_in_01, traffic_out_23`).
* `<условия>`: Комбинация условий через `AND`.
    * `<имя_поля> <оператор> <значение>`
    * Поддерживаемые поля для `WHERE`: `name`, `ip`, `date`.
    * Операторы:
        * Для `name` (строка): `=`, `!=`, `CONTAINS`. Значение в двойных кавычках.
        * Для `ip` (IP-адрес): `=`, `!=`, `<`, `>`, `<=`, `>=`. Значение IP-адреса в двойных кавычках.
        * Для `date` (дата ДД/ММ/ГГГГ): `=`, `!=`, `<`, `>`, `<=`, `>=`. Значение даты в двойных кавычках.
* `<поле_сортировки>`: `name`, `ip`, `date`, `total_traffic`, `total_incoming`, `total_outgoing`, `traffic_XX`, `traffic_in_XX`, `traffic_out_XX`.
* `ASC` (по умолчанию) / `DESC`: Порядок сортировки.

**Примеры:**
* `SELECT *`
* `SELECT name, ip WHERE date >= "01/01/2024"`
* `SELECT total_traffic, name WHERE name CONTAINS "Doe" SORT BY total_traffic DESC`

### 2. `ADD`
`ADD name "<ФИО>" ip "<IP>" date "<Дата>" traffic "<трафик_по_часам>"`
* `<трафик_по_часам>`: Строка из 48 чисел (вх0 исх0 вх1 исх1 ... вх23 исх23), разделенных пробелами, в двойных кавычках.

**Пример:**
* `ADD name "New User" ip "10.20.30.40" date "15/05/2024" traffic "10 20 0 0 ..."`

### 3. `DELETE`
`DELETE WHERE <условия>`
* `<условия>`: Такие же, как для `SELECT WHERE`.

**Пример:**
* `DELETE WHERE name = "Old User" AND ip = "1.2.3.4"`

### 4. `EDIT` (через `RAW_EDIT_COMMAND` в пакетном режиме клиента)
`RAW_EDIT_COMMAND key_name "<тек_ФИО>" key_ip "<тек_IP>" key_date "<тек_Дата>"###SEP###[set_name "<нов_ФИО>"] [set_ip "<нов_IP>"] [set_date "<нов_Дата>"] [set_traffic "<нов_трафик>"]`
* `key_*`: Обязательные поля для идентификации записи.
* `###SEP###`: Обязательный разделитель.
* `set_*`: Опциональные поля для установки новых значений. Если поле `set_*` отсутствует, соответствующий атрибут записи не изменяется.

**Пример:**
* `RAW_EDIT_COMMAND key_name "User A" key_ip "1.1.1.1" key_date "01/01/2024"###SEP###set_ip "1.1.1.2" set_name "User Alpha"`

### 5. `CALCULATE_BILL`
`CALCULATE_BILL <начальная_дата> <конечная_дата> [WHERE <условия>]`
* Даты в формате ДД/ММ/ГГГГ.
* `<условия>`: Такие же, как для `SELECT WHERE`. Если отсутствуют, расчет для всех записей в указанном периоде.

**Пример:**
* `CALCULATE_BILL 01/01/2024 31/01/2024 WHERE name = "Alice Wonderland"`

### 6. `LOAD_TARIFF`
`LOAD_TARIFF <имя_файла_тарифа>`
* Загружает новый тарифный план из указанного файла. На сервере файл ищется относительно директории основного файла тарифов.

**Пример:**
* `LOAD_TARIFF tariff_alt.cfg`

## Кастомный Сетевой Протокол

Взаимодействие между `db_client_app` и `db_server_app` происходит по простому текстовому протоколу поверх TCP/IP.
* **Обрамление сообщений:** Каждое сообщение (запрос или ответ) обрамляется префиксом длины (8 цифр), указывающим на длину последующей полезной нагрузки.
* **Структура полезной нагрузки:**
    1.  Числовой идентификатор типа команды (`CommandType` из `network_protocol.h`).
    2.  Символ новой строки (`\n`).
    3.  Фактические данные команды/ответа (например, строка запроса `SELECT...` или результат).
* **Типы команд (`CommandType`):** `PING`, `EXECUTE_QUERY` (для SELECT, ADD, DELETE, CALCULATE_BILL, LOAD_TARIFF), `SHUTDOWN_SERVER_CMD`, `EDIT_RECORD_CMD` и типы ответов (`ERROR_RESPONSE`, `SUCCESS_RESPONSE_DATA`, `SUCCESS_RESPONSE_NO_DATA`, `ACK_PONG`).

## HTTP Интерфейс

Сервер `db_server_app` также предоставляет HTTP интерфейс для взаимодействия с базой данных через браузер.
* **Основной URL:** `http://<хост_сервера>:<порт_сервера>/` - отображает HTML-форму со всеми доступными операциями.
* **Точка обработки запросов:** `http://<хост_сервера>:<порт_сервера>/query`
* **Поддерживаемые команды (через параметр `command` в POST или `query` в GET):**
    * **`SELECT` (GET):**
        * Метод: `GET`
        * URL: `/query?query=<полный_SELECT_запрос_url_encoded>`
    * **`ADD` (POST):**
        * Метод: `POST`
        * Параметры формы: `command=add`, `name=<ФИО>`, `ip=<IP>`, `date=<Дата>`, `traffic=<трафик_строка>`
    * **`EDIT` (POST):**
        * Метод: `POST`
        * Параметры формы: `command=edit`, `key_name=<тек_ФИО>`, `key_ip=<тек_IP>`, `key_date=<тек_Дата>`, `set_name=[<нов_ФИО>]`, `set_ip=[<нов_IP>]`, `set_date=[<нов_Дата>]`, `set_traffic=[<нов_трафик_строка>]` (поля `set_*` опциональны).
    * **`DELETE` (POST):**
        * Метод: `POST`
        * Параметры формы: `command=delete`, `conditions=<условия_WHERE>`
    * **`CALCULATE_BILL` (POST):**
        * Метод: `POST`
        * Параметры формы: `command=calculate_bill`, `start_date=<ДД/ММ/ГГГГ>`, `end_date=<ДД/ММ/ГГГГ>`, `bill_query_conditions=[<условия_WHERE>]`
    * **`LOAD_TARIFF` (POST):**
        * Метод: `POST`
        * Параметры формы: `command=load_tariff`, `tariff_filename_http=<имя_файла.cfg>` (файл ищется на сервере в директории основного тарифа).
* Ответы сервера представляются в виде HTML-страниц.

## Тестирование

Проект включает два уровня тестирования:

### 1. Юнит-тесты
* Находятся в директории `tests/unit/`.
* Используют фреймворк Google Test.
* Тестируют отдельные классы и модули (`Database`, `QueryParser`, `ProviderRecord`, `TariffPlan`, `Date`, `IpAddress`, `common_defs`).
* Запускаются командой `ctest` (или `make test` / `ninja test`) из директории сборки после компиляции.

### 2. Системные (End-to-End) тесты
* Находятся в директории `tests/system/`.
* Реализованы в виде shell-скриптов (`.sh`), которые запускают собранные приложения с различными параметрами и данными.
* Сравнивают фактический вывод и состояние файлов БД с ожидаемыми.
* Запускаются командой `ctest` (или `make test` / `ninja test`).
* **Ключевые системные тесты:**
    * `run_standalone_batch_test.sh`: Тестирование пакетного режима монолитного приложения.
    * `run_client_server_single_test.sh`: Тестирование взаимодействия одного клиента с сервером, включая все основные команды.
    * `run_client_server_multi_test.sh`: Параметризованный скрипт для запуска различных сценариев (A, B, C, D, E, F) с тремя клиентами для проверки конкурентного доступа:
        * **A:** Конкурентное редактирование (имени) и последующее удаление одной записи.
        * **B:** Конкурентное чтение во время многократных операций записи в одну запись.
        * **C:** Конкурентная загрузка тарифа одним клиентом и расчет счетов другими.
        * **D:** Конкурентное редактирование разных атрибутов одной записи разными клиентами.
        * **E:** Конкурентное удаление одной и той же записи.
        * **F:** Отправка команды `SHUTDOWN_SERVER` во время активных операций других клиентов.
    * `run_http_tests.sh`: Тестирование HTTP-интерфейса сервера с помощью `curl` для всех поддерживаемых команд (SELECT, ADD, EDIT, DELETE, CALCULATE_BILL, LOAD_TARIFF), включая проверки на ошибки и безопасность.
